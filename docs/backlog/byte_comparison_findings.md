# Byte-Level Comparison: Rust vs Java Indexes

## Test Setup

- **Corpus**: 500 documents generated by `testdata/gen_docs.py -n 500` (35.9 MB synthetic text)
- **Indexing**: Single-threaded, non-compound, single segment (`_0`)
- **Java**: Lucene 10.3.2 via `IndexAllFields`, stored field "notes" = "indexed by Java"
- **Rust**: Bearing `indexfiles`, stored field "notes" = "indexed by Rust"
- **Comparison tool**: `/tmp/compare_indexes.py` â€” masks 16-byte segment ID in codec headers and 8-byte CRC in codec footers before comparing

Every Lucene codec file contains a header with a random 16-byte segment ID and a footer with a CRC32 checksum over the file contents. These differ between any two indexing runs, even two Java runs. The comparison tool masks both before comparing payload bytes.

### Same size, differ â€” verified root cause (6 files)

| File | Size | Root Cause |
|------|-----:|-----------|
| `_0.fnm` | 936 | Per-field attribute map iteration order differs (Java `HashMap` vs Rust `HashMap`). Both contain identical key-value pairs. Unfixable without sorting, and Java doesn't sort. |
| `_0.fdm` | 158 | Stored fields metadata. 1 byte differs at offset 0x6e â€” a pointer into `.fdt`, which differs due to LZ4 compression of different content. Verified by byte inspection. |
| `_0.kdd` | 8,293 | BKD points data. Field write order differs â€” Java iterates `fieldHash[]` (hash table order), Rust iterates by ascending field number. This reorders per-field data blocks but per-field content is otherwise identical. (Low cardinality VInt off-by-one was fixed â€” Rust now writes `VInt(cardinality)` matching Java.) |
| `_0.kdi` | 74 | BKD points index. Per-field data is identical except VLong-encoded file pointers differ due to field write order in `.kdd` (different dataFP values encode to different VLong sizes). |
| `_0.kdm` | 306 | BKD points metadata. Field write order differs (hash table vs field number order), causing field numbers and file pointers to appear in different sequence. Per-field metadata content is identical. |
| `segments_1` | 155 | Segment commit metadata. Three random 16-byte IDs differ every run (header ID, segment ID, SCI ID), plus CRC. One non-ID difference: `version` field (Java=4, Rust=1) â€” an opaque monotonic counter incremented by Java's `IndexWriter` lifecycle methods; has no semantic meaning to readers. All other fields identical. Verified by field-by-field parsing. |
